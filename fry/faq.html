Frequently asked questions

Why another language?

In Simon Peyton Jones's taxonomy of programming language lifecycles, the projected future of Fry is the quick death:
So why bother?
  • Fry is an experiment in declarative object-oriented programming. Note that it is not meant to be a multi-paradigm language. Let me explain.

    Functional languages have a number of distinctive features. These are:

    • first-class lambda expressions,
    • expression-based programming constructs (composition, preferably both functional and monadic),
    • (usually non-mutable) algebraic datatypes with pattern matching.

    Moreover, and more importantly, functional programming is a philosophy of programming, the way you think about a program. This means that (in contradiction to what you can read on Wikipedia), Java and C# are not functional languages, though they provide first-class lambda expressions!

    The same holds for object-oriented (OO) languages, though it is harder to pinpoint the distinctive features, as there are a number of sub-genres. Let's take Ruby as an example. Its OO features are:

    • objects,
    • mutable state,
    • classes with subtyping,
    • inheritance (as tool for implementation-sharing; the subtyping hierarchy can be disjoint from the inheritance hierachy).

    Some people, like yours truly, are very fond of the elegance and expressiveness of functional languages, so they wish to share it with the OO community by creating multi-paradigm languages, which have the OO features plus the functional features. This creates a number of philosphical issues, because some of the OO features solve problems in a different way than functional features; take, for example, dynamic dispatch vs pattern matching.

    The philosophy of Fry is to take only some features of functional languages and some of OO languages. We take dynamic dispatch, but no pattern matching; expression-based programming, but no mutable state. The case that Fry tries to support is that object-oriented programming is not necessarily imperative.

  • We need simple programming languages, and object-oriented languages tend to resemble the Kafkian bureaucracy: few have a real grasp of the enormity of the language and its semantics. XXX, the creator of Ruby, said: "Ruby is not simple, but it is notural". Not good enough! We want to do better! We can afford simple programming languages!

    Easier say than done, one might accuse Fry, as it does not have all the features of a fully fledged programimng language: no modules, no strong type system, no support for concurrency, and so on. True, but these features are orthogonal to the problem in question, and these are not the sole reason why modern-day languages are so complicated.

  • The simplicity of Fry makes it interesting pedagogically. It distils the concept of object-oriented programming, and shows that it is orthogonal to imperative programming. Also, Fry's simple semantics makes it a good candidate to study semantics and implementation of programming languages.

Why dynamic types?

In reality, Fry is a functional language in disguise, and strong static type systems go well with functional languages. The reason why Fry is not strongly-typed is merely its under-development. This means that Fry may become strongly-typed at some point in the future.

Why the name?

Fry is a character in the TV show Futurama, who happens to be his own grandson (en.wikipedia.org/wiki/Roswell_That_Ends_Well). It reflects the concept of circular inheritance, which is a distinctive feature of Fry.
